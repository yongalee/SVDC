Abstract

This document specifies the software design for a Sampled Value Data Concentrator (SVDC), the central data-plane component of an a²SDP local node as described in Protection and Control of Active Distribution Systems (Meliopoulos et al., CIGRE 2024, Paper ID 10427). The proposed implementation captures IEC 61850-9-2 Sampled Value (SV) streams from multiple Merging Units, time-aligns them under PTP-disciplined timestamps, buffers them in dual redundant circular buffers, and exposes a pull-based subscriber API to all node-local applications including Estimation-Based Protection (EBP) relays, the Phasor Computation Module, the Transient Recorder, and the Fault Locator. The design further realizes the write-back path required for self-healing replacement of compromised SVs identified by the node-level Quasi-dynamic State Estimator (QSE).

The work is offered by Shinsung Industrial Electric (SSIEC) as a software contribution to the a²SDP research programme, intended for use at Prof. Meliopoulos's existing US partner utility deployments (Southern Company, Avista, Dominion). SSIEC's role is that of an external software contributor; the deliverable is the SVDC implementation itself, packaged for installation and operation by the Georgia Tech team.

§ 1Background and motivation

1.1The a²SDP local node

The a²SDP architecture partitions an Active Distribution System into sections, each managed by a local node consisting of a GPS-disciplined PTP grandmaster, a 1 Gbps Ethernet switch, a multicore compute platform, and an uninterruptible power supply [1]. The node ingests Sampled Values (SV) from dispersed Merging Units (MUs) over dual single-mode fiber, runs a population of Estimation-Based Protection (EBP) relays — one per protection zone — and synthesizes a node-level Quasi-dynamic State Estimate (QSE) that is forwarded to the master node via IEEE C37.118.

1.2The role of the SVDC

The SVDC sits between the MU population (south) and the application population (north). It absorbs the asynchronous arrival of SV packets — synchronized in timestamp to better than one microsecond, but received with up to two milliseconds of variable latency owing to fiber length, queueing, and switch traversal — and presents to every application a coherent, time-aligned, gap-free SV stream organized in dual redundant circular buffers. This decoupling is the foundation on which all node-local applications operate.

1.3Why a clean reference implementation matters

In existing field deployments the SVDC is typically vendor-bundled with proprietary protection platforms, which complicates the integration of new applications, the addition of new MU vendors, and the introduction of the self-healing write-back path described in [1]. An open, independent SVDC implementation aligned strictly to the a²SDP specification offers three benefits to the research programme:

  • It removes vendor-coupling from the data plane, enabling new applications and new MU vendors to be onboarded against a stable specification.
  • It provides a reference against which factory and field test procedures (per [1, §10]) can be standardized across pilot sites.
  • It serves as a clean, independently maintained substrate that the Georgia Tech team can deploy at any current or future a²SDP partner utility site without negotiating with an incumbent vendor's protection platform.

§ 2Scope

2.1In scope

  • Ingestion of IEC 61850-9-2 (LE profile) Sampled Value streams over Ethernet, with PTP-disciplined timestamp interpretation.
  • Time alignment of SVs across an arbitrary number of MUs registered to the node.
  • Dual redundant circular buffer management.
  • Interpolation of occasional lost samples per [1, §7(f)].
  • Pull-based subscriber API for node-local applications (EBP relays, Transient Recorder, Phasor Computation Module, Fault Locator).
  • Write-back API enabling the QSE to overwrite compromised SVs with state-estimated replacements per [1, §7] and Figure 4.
  • Health and diagnostic surfaces required by the factory and field test procedures of [1, §10].
  • A reference MU simulator sufficient for end-to-end functional verification in the absence of physical MUs.

2.2Out of scope

  • The EBP relay engine itself (consumer of SVDC data).
  • The QSE algorithm itself (consumer and write-back producer).
  • The PTP grandmaster and slave clock implementations (assumed provided by hardware or by linuxptp).
  • The MU hardware and its analog front-end calibration.
  • Network switch configuration and VLAN/multicast provisioning.
  • The master-node Phasor Data Concentrator (a distinct component receiving IEEE C37.118 streams).

§ 3Functional requirements

The functional surface of the SVDC is derived directly from [1, §3, §6, §7]. Each requirement below is traceable to a paper-level statement.

IDRequirement
FR-1 SV packet ingestion. The SVDC shall subscribe to IEC 61850-9-2 SV multicast groups on the process-bus interface and decode all configured ASDUs.
FR-2 Time alignment. The SVDC shall group decoded samples by their PTP-disciplined timestamp into per-tick records covering all registered MU channels.
FR-3 Dual redundant circular buffers. The SVDC shall maintain two independent circular buffers (CB-A, CB-B), each holding the most recent N aligned records, where N is configurable and shall default to a window sufficient for at least 64 power-frequency cycles.
FR-4 Gap interpolation. When an expected sample from a registered MU channel fails to arrive within the alignment deadline, the SVDC shall insert an interpolated value derived from the temporally adjacent valid samples and flag the record as INTERPOLATED.
FR-5 Concurrent subscribers. The SVDC shall support concurrent read access by an arbitrary number of subscribing applications, each selecting an independent channel set and a sliding time window, without blocking the ingest path.
FR-6 Self-healing write-back. The SVDC shall accept overwrite requests from the QSE that replace identified compromised samples with estimated values, preserving the original samples in an audit log and flagging the affected records as QSE_CORRECTED.
FR-7 Channel registry. The SVDC shall maintain a runtime registry of MUs and their channels, including phase identity, instrument transformer ratio, polarity, and applied calibration factors per [1, §10].
FR-8 Health surfaces. The SVDC shall expose, per channel and per MU, observed sample rate, observed arrival jitter, missing-sample count, and last applied calibration factor.
FR-9 Configuration via SCD. The SVDC shall accept its MU and channel configuration from an IEC 61850 SCL file (SCD) and shall not require recompilation for configuration changes.
FR-10 Determinism. The SVDC shall guarantee that every successfully ingested SV is visible to subscribers within a bounded latency from its arrival at the process-bus interface (see NFR-2).

§ 4Non-functional requirements

IDTarget
NFR-1 Timestamp accuracy. Honour the <1 µs PTP-disciplined synchronization established by the merging units; the SVDC itself shall add no timestamp distortion exceeding 100 ns end-to-end.
NFR-2 Internal latency. Median latency from packet reception on the NIC to availability in a circular buffer slot shall not exceed 200 µs at full load; 99th-percentile shall not exceed 500 µs. This budget sits well inside the <2 ms total SV latency reported in [1, §3].
NFR-3 Throughput. Sustain the full sample-rate output of up to 32 MUs publishing 80 samples/cycle (60 Hz) on each of 8 analog channels, with headroom of ≥2× for burst handling.
NFR-4 Memory residency. All hot-path buffers shall be statically allocated and pre-touched at startup; no allocation, no GC, no page-faulting in the ingest or alignment paths.
NFR-5 CPU isolation. The ingest and alignment threads shall be pinnable to dedicated cores via cpuset/isolcpus; the SVDC shall expose configuration for this.
NFR-6 Subscriber non-interference. A slow or stalled subscriber shall not impede ingest or any other subscriber. Read consistency is per-snapshot; back-pressure is not propagated to the ingest path.
NFR-7 Recoverability. Loss of one circular buffer shall not interrupt service; the surviving buffer continues to satisfy subscribers while the failed buffer is rebuilt.
NFR-8 Observability. All counters and health metrics shall be exposed via a structured (JSON-over-HTTP) endpoint for both the QSE and external monitoring.
NFR-9 Portability. The SVDC shall build and run on Ubuntu 22.04 LTS or later with the PREEMPT_RT patchset, and the build shall produce no platform-specific binaries beyond the kernel driver layer.
NFR-10 Language convention. All user-facing artefacts shall be in English without exception: CLI option names and help text, log messages, error messages, HTTP/JSON field names and response payloads, configuration-file keys, source-code comments, identifier names, commit messages, and operator documentation. This convention supports international code review with the Georgia Tech team, multi-utility pilot deployment, and unambiguous integration with the reference QSE.

§ 5System architecture

5.1Context

Figure 1 places the SVDC in the local-node context defined by [1, Figures 2 and 4]. Southbound it terminates SV streams from the MU population. Northbound it serves EBP relays, the Phasor Computation Module (which feeds the Virtual Phasor Data Concentrator and onward to the master node via IEEE C37.118), the Transient Recorder, and the Fault Locator. The QSE both subscribes (for its inputs) and writes back (for self-healing corrections).

Figure 1   SVDC in the a²SDP local-node context

5.2Internal module decomposition

The SVDC decomposes into eight cohesive modules with sharply scoped responsibilities. The ingest and alignment paths are lock-free and pinned to isolated cores; the subscriber and write-back paths use snapshot semantics on the circular buffers and require no mutexes on the hot path.

§ 6Component design

M1 · Ingress

SV reception and decoding

Bound to the process-bus NIC via AF_PACKET (Linux) with a kernel-bypass option for DPDK in a later iteration. Decodes IEC 61850-9-2 LE ASDUs into a fixed-layout internal record. Extracts the PTP-disciplined sample count and converts to a 64-bit nanosecond UTC timestamp using the node's reference clock. No per-packet allocation: a pre-allocated pool of record slots is recycled in a SPSC ring to M2.

M2 · Time Aligner

Per-tick record assembly

Maintains a sliding alignment window keyed by sample-count tick. Per registered MU channel, places the decoded sample into the appropriate tick slot. When all expected channels have reported for a given tick, OR when the alignment deadline (configurable, default 1.5 ms past the tick's nominal time) expires, the completed tick record is forwarded to M3.

M3 · Interpolator

Gap fill

For tick records that exit the aligner with any missing channel, the missing values are filled by linear interpolation between the most recent valid sample preceding the gap and the first valid sample following the gap (when available; otherwise by zero-order hold). The affected sample is flagged INTERPOLATED with the gap length recorded for downstream visibility.

M4 · Calibration

Channel calibration application

Applies the per-channel calibration triple (scale, offset, φ) derived from the procedures of [1, §10]. Magnitude correction is multiplicative, DC offset correction is additive, and timing correction φ is realized via fractional-sample re-interpolation across the aligned window. Calibration factors are updated atomically through the management interface; the application path reads them via a versioned snapshot.

M5 / M6 · Dual Circular Buffers

Lock-free SPMC ring storage

Two independent circular buffers, each holding N fully-aligned, calibrated tick records. The producer (M4) writes a record by reserving the next slot, populating it, and incrementing the published cursor with release semantics; readers (M7) take a snapshot of the cursor with acquire semantics and walk backward. CB-A is the primary; CB-B receives the same write stream and is structurally identical. On detected corruption of CB-A (e.g., checksum mismatch on snapshot), reads transparently fall over to CB-B while CB-A is rebuilt from CB-B's contents.

M7 · Subscriber API

Pull-based application access

Each subscriber registers with a channel set and receives a cursor. Read calls return either (a) the most recent k records covering that channel set, or (b) all records since the subscriber's last cursor position. Records are zero-copy where the subscriber is in-process; cross-process subscribers receive serialized records over a UNIX domain socket. A slow subscriber's cursor falls behind without affecting any other party.

M8 · Write-back

QSE-driven self-healing path

Accepts overwrite requests of the form (channel_id, tick_id, estimated_value, qse_quality). The target record's original value is copied to the audit log, the slot is updated in place in both CB-A and CB-B, and the record flag is set to QSE_CORRECTED. The write-back path is rate-limited and reject-fast: if the requested tick has already aged out of the buffer, the request is rejected with an explicit error. Subscribers reading a corrected record see the new value and the corrected flag; the original is preserved in the audit log for offline analysis.

M9 · Health & Diagnostics

Telemetry surface

Per-MU and per-channel metrics: observed sample rate, arrival jitter histogram, missing-sample count, interpolation count, QSE-correction count, PTP synchronization status, applied calibration triple. Exposed over a structured JSON-over-HTTP endpoint and via a streaming subscription for the QSE's data-validation module. These surfaces directly support the test procedures of [1, §10].

§ 7Data model

7.1The aligned tick record

The internal record assembled by M2 and stored in M5/M6 is laid out for cache-line alignment and for fast subscriber walks. The wire-domain SV format is decoded into this canonical representation once on ingest.

struct TickRecord {
    u64     tick_id;          // monotonic per-node tick counter
    u64     ts_utc_ns;        // PTP-disciplined UTC nanoseconds
    u16     n_channels;       // number of populated channels
    u16     flags;            // COMPLETE | INTERPOLATED | QSE_CORRECTED | DEGRADED
    u32     crc;              // integrity check across samples[]
    Sample  samples[MAX_CH];  // indexed by channel_id in channel registry
};

struct Sample {
    i32     value_q;          // calibrated value, Q-format scaled to channel unit
    u8      quality;          // IEC 61850 quality bits
    u8      origin;           // LIVE | INTERPOLATED | QSE_ESTIMATED
    u16     reserved;
};

7.2Channel registry

The runtime channel registry is loaded from the SCD file at startup and exposes:

§ 8External interfaces

8.1Southbound — SV ingest

IEC 61850-9-2 LE multicast over Ethernet. The SVDC subscribes to the multicast MAC addresses listed in the SCD; no application-level handshake. PTP (IEEE 1588 / IEC 61588) timestamps are assumed disciplined by an external linuxptp instance and consumed via SO_TIMESTAMPING.

8.2Northbound — Subscriber API

Two transport bindings, identical semantics:

  • In-process (C ABI): zero-copy access for performance-critical consumers (EBP relays, Phasor Computation Module).
  • Out-of-process (UNIX domain socket, length-prefixed framing): for consumers in separate processes or other languages.
// In-process API sketch
SvdcCursor  svdc_subscribe(const ChannelSet* cs);
int         svdc_read_latest(SvdcCursor, size_t k, TickRecord** out);
int         svdc_read_since(SvdcCursor, TickRecord** out, size_t* n_out);
void        svdc_release(SvdcCursor, TickRecord**);
void        svdc_unsubscribe(SvdcCursor);

8.3Write-back — QSE correction

struct QseCorrection {
    u16  channel_id;
    u64  tick_id;        // must lie within the current CB window
    i32  estimated_value_q;
    u8   qse_quality;
};

int svdc_writeback(const QseCorrection* batch, size_t n);
// returns 0 on success; per-element error codes via out-parameter

8.4Management & telemetry

HTTP/JSON endpoint on a configurable port: GET /health, GET /channels, GET /metrics, POST /calibration/{channel_id}. The calibration endpoint is the runtime surface for the field-monitoring procedure of [1, §10] (Figure 11 in the paper).

Note on Rust

A Kotlin/JVM reference implementation is feasible on the strength of SSIEC's existing IEC 61850 codebase, but JVM pause behaviour conflicts with NFR-2/4. Rust is recommended for the production-grade reference; a Kotlin port of the subscriber API can be supplied for consumers already on the JVM.

§ 10Testing & validation

10.1Test layers

  • Unit tests on every module, with a property-based suite for M2 (Time Aligner) and M5/M6 (CBs).
  • MU simulator harness that synthesizes SV streams from an arbitrary number of virtual MUs, with configurable jitter, loss rate, and constant timing/magnitude/DC-offset injection. This directly mirrors the laboratory setup of [1, §10] (paper Figures 7, 9).
  • Loopback integration tests exercising ingest → alignment → CB → subscriber, with assertions on tick completeness, ordering, and per-record flags.
  • Latency & jitter benchmarks against NFR-2/3, with histograms produced in the style of paper Figure 8.
  • QSE write-back tests driven by a stub QSE, verifying audit-log preservation, flag correctness, and rejection of out-of-window corrections.
  • Long-running soak (≥ 24 h) on the laboratory hardware, equivalent in intent to the 18.8-minute synchronization soak of paper Figure 8 but extended in duration.

§ 13Collaboration model

The proposed collaboration is asymmetric and explicit:

  • SSIEC contributes: the SVDC implementation, the conformance-grade SV Publisher reference simulator (ssiec-sv-publisher), the test suite, and ongoing engineering support for integration at the Georgia Tech US partner utility sites.
  • Georgia Tech contributes: the a²SDP specification authority (acceptance review on FR/NFR interpretation), access to a reference QSE for integration testing, and the option to host the repository under the a²SDP programme's organization.
  • IP and licensing: Apache-2.0 source; both parties retain freedom to use, extend, and redistribute. Patentable inventions arising from the implementation work (e.g., a novel lock-free interleaving scheme) are owned by the inventing party with a no-cost cross-license to the other.
  • Cadence: bi-weekly working sync (video); milestone-gated demo reviews; asynchronous code review on GitHub.
  • Publication path: a joint paper at CIGRE 2026 (Paris Session) is proposed as the natural venue for the integration results, with optional submission to IEEE Transactions on Power Delivery for the architectural treatment.

§ 15Open questions for the spec-lock review

  1. What is the intended default value of N (records held per CB), and the constraints that drive it? The paper specifies "N most recent values" without numeric bound.
  2. What sample rate is canonical for the reference deployment — 80 SPC (protection) or 256 SPC (measurement) per 60 Hz cycle? Different sample rates have downstream consequences for the Phasor Computation Module.
  3. For interpolation, is linear sufficient, or is a higher-order method expected (e.g., the quadratic fractional sample integration mentioned in [1, §7] for the Standard-PMU algorithm)?
  4. Should the write-back path be authenticated/authorized at the API level (e.g., only the local QSE may write), or is process-level isolation considered sufficient?
  5. Is there an existing reference SCD schema for an a²SDP node that the SVDC should consume, or should one be defined as part of M0?
  6. What is the preferred mechanism for clock-holdover behaviour reporting from linuxptp into the SVDC's health surface?
