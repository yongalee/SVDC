Abstract

This document operationalizes the SVDC Software Design Document v0.1 into a step-by-step development plan. Engineering work is decomposed under the MECE principle (mutually exclusive, collectively exhaustive) into eight Work Breakdown Structure (WBS) categories with approximately fifty individual work items, each traceable to a functional requirement, non-functional requirement, or design module of the parent document.

The WBS is then sequenced into seven execution phases (Phase 0 through Phase 6) running across a twelve-week schedule. Each phase carries a single goal, an explicit WBS scope, a concrete task checklist, and measurable exit criteria. Three formal Quality Gates (G0, G3, G6) mark the points at which SSIEC requests review and sign-off from the Georgia Tech a²SDP programme.

§ 1Purpose and methodology

1.1Relation to the Software Design Document

The Software Design Document v0.1 (SDD) specifies what the SVDC shall be: its functional and non-functional requirements, its module decomposition (M1–M9), its data model, and its external interfaces. This Implementation Plan specifies how SSIEC will deliver it: the discrete work items, their sequencing, the engineering effort required, and the checkpoints at which Georgia Tech is asked to review.

Every WBS item in §2 below carries a traceability tag pointing back to one or more FRs, NFRs, or design modules of the SDD. Conversely, every FR/NFR/module of the SDD is covered by at least one WBS item — this is the collective-exhaustiveness half of MECE.

1.2MECE decomposition

Work is partitioned into eight top-level WBS categories, chosen so that each work item belongs to exactly one category (mutually exclusive) and so that the categories together cover all engineering effort needed for a v0.1 release (collectively exhaustive):

  • WBS-1 · Project Foundation — repository, build system, CI, coding standards.
  • WBS-2 · Core Data Plane — the hot path: ingest, decode, align, interpolate, calibrate, buffer.
  • WBS-3 · External Interfaces — subscriber API, write-back API, management endpoint.
  • WBS-4 · Configuration and Registry — SCD parser, channel registry, runtime calibration management.
  • WBS-5 · Observability — metrics, telemetry, structured logging, audit log.
  • WBS-6 · Test Infrastructure — MU simulator, stubs, harnesses, fault injection.
  • WBS-7 · Validation — unit, property, integration, performance, soak, acceptance.
  • WBS-8 · Documentation and Release — API reference, integration guide, performance report, packaging.

1.3Phased execution

The MECE WBS answers "what work exists." It does not by itself answer "in what order." §4 sequences the WBS into seven phases that respect technical dependencies (e.g., subscribers cannot be tested before buffers exist), align demos to natural cohesive units of progress, and place the three formal Quality Gates at points where the deliverable is independently meaningful to Georgia Tech.

1.4Northbound interface strategy

v0.1 of this plan inherited the SDD's assumption that node-local applications consume SV data via an in-process C ABI or a UNIX domain socket. That assumption is correct for the sub-millisecond protection path (EBP relays, Phasor Computation Module, QSE) but is insufficient for the broader operational context where SCADA, HMI, asset management, historian, and cloud analytics systems also need access to SVDC data — typically over standards already operating at the host utility.

v0.2 therefore adopts a layered northbound architecture. Each layer targets a distinct class of consumer with the interface that consumer expects, and the layers are isolated so that failure in one (e.g., a stalled historian) cannot impair the others (e.g., the protection path). This is the dominant pattern in current industrial data architecture practice.

LayerTarget consumerInterfaceRationale
L0 Same-node protection apps: EBP relays, Phasor Computation Module, QSE In-process C ABI + shared memory (CB direct read); UDS framing for cross-process consumers in the same node Sub-millisecond latency, zero-copy. Unchanged from SDD §8.2.
L1 SCADA, HMI, asset management, plant-wide OT systems OPC UA Server exposing channels and aligned records per OPC Foundation's IEC 61850 companion specification (OPC 10040) OPC UA is the industrial de-facto standard (IEC 62541). The IEC 61850 → OPC UA mapping is itself standardized, so the channel registry maps automatically into the OPC UA Information Model.
L2 Analytics, ML training, cloud, multi-consumer fan-out MQTT publisher (rumqttc) on a configurable broker; payload reuses OPC UA encoding (PubSub over MQTT pattern) Temporal decoupling, fan-out to arbitrary subscribers without coupling to SVDC. Quality of service tunable per consumer class.
L3 Historian, replay, audit, BI tools TimescaleDB sidecar — separate process that subscribes to L2 MQTT and persists into a time-series schema DB write load is fully isolated from the hot path. TimescaleDB chosen for PostgreSQL compatibility (existing BI/SQL tools work unchanged).
Design principle · failure isolation

L0 has no dependency on L1, L2, or L3. L1 has no dependency on L2 or L3. L2 has no dependency on L3. A failure in any layer is contained to that layer and the layers above it; the protection path (L0) survives any northbound failure. The TimescaleDB sidecar in particular runs in a separate OS process and on a separate CPU set from the SVDC core.

What this changes in the plan: WBS-3 grows to include the L1 OPC UA Server (new WBS-3.7), the L2 MQTT publisher (new WBS-3.8), and the L3 TimescaleDB sidecar (new WBS-3.9). Phase 4 in §4 absorbs the L1/L2 integration work, lengthening Phase 4 from one week to two weeks. Total effort rises from 89 PD to 102 PD; total schedule from 12 to 13 weeks.

§ 4Phased execution plan

Each phase below is presented as a self-contained execution card. The phase begins only when the prior phase's exit criteria pass, and ends only when its own exit criteria pass. Within a phase, tasks may be parallelized across engineers but should not span a phase boundary without explicit re-planning.

§ 5Quality gates

Three formal Quality Gates structure the engagement with Georgia Tech. Between gates, communication is asynchronous (bi-weekly written status + GitHub PR reviews). Each gate requires explicit sign-off before the next phase begins.

GateNameTriggered afterSign-off requires
G0 Spec Lock Phase 0 Professor's written decisions on the six §15 open questions; SSIEC's commitment to scope baseline. Both committed to docs/spec-lock-v0.1.md.
G3 Mid-project Demo Phase 3 Recorded demo of full data plane with multi-subscriber + write-back. Professor's review comments addressed or logged as deferred. Optional: 30-minute live walkthrough.
G6 v0.1 Release Phase 6 All §10.2 acceptance criteria signed off. Performance report delivered. Georgia Tech confirms ability to integrate with at least one worked example. v0.1.0 GitHub release published.
On gate failure

If a gate fails, the affected phase is reopened with a remediation plan committed to the repo. The schedule slips by the remediation duration; downstream phases are re-baselined. Gates are not waived.

Documentation discipline

Every gate review, every spec clarification, every accepted scope change is captured in the repo under docs/decisions/ as a dated ADR (Architecture Decision Record). The repo is the single source of truth; no decision exists if it is not in the repo.
